Exercise 4 — Branching, Merging & .gitignore

Module 3 | ~40 minutes

Goal

Try an alternative analysis approach on a branch, merge it back, resolve a realistic merge conflict, and exclude raw data files from tracking.

Continues the experiment-tracker project from Exercise 3.

Part A — Create a branch for a new approach

cd ~/Downloads/experiment-tracker

git branch                              # which branch are you on?
git checkout -b try-new-normalization   # create and switch
git branch                              # confirm

Part B — Make changes on the branch

nano analysis_log.md

Append:

## Step 2b — Alternative: quantile normalization
Tried quantile normalization instead of log2. Compare results before merging.
git add analysis_log.md
git commit -m "tried quantile normalization instead of log2"
git log --oneline

Part C — Merge back to main

git checkout main
cat analysis_log.md      # the quantile normalization note is NOT here

git merge try-new-normalization
cat analysis_log.md      # now it IS here

git log --oneline
git branch -d try-new-normalization    # clean up

Part D — Simulate a merge conflict

git checkout -b draft-conclusion
nano results_summary.md

Write:

# Results Summary
Treated samples show a 2-fold increase in expression compared to control.
git add results_summary.md
git commit -m "drafted conclusion: 2-fold increase"

git checkout main
nano results_summary.md

Write a different conclusion on main:

# Results Summary
Treated samples show no significant change in expression compared to control.
git add results_summary.md
git commit -m "drafted conclusion: no significant change"

git merge draft-conclusion    # CONFLICT

Open the file and find the conflict markers:

<<<<<<< HEAD
Treated samples show no significant change in expression compared to control.
=======
Treated samples show a 2-fold increase in expression compared to control.
>>>>>>> draft-conclusion

Edit to keep the conclusion you actually want (or merge both into one sentence noting the discrepancy needs investigation). Remove all marker lines, then:

git add results_summary.md
git commit -m "resolved conflicting conclusions, flagged for re-analysis"

Check your understanding

Question: Why did git merge try-new-normalization in Part C succeed silently, but git merge draft-conclusion in Part D stopped with a conflict?

TipClick to reveal the answer

In Part C, only the try-new-normalization branch had changed analysis_log.md — main hadn’t touched that file since they diverged, so Git could apply the change cleanly (a “fast-forward” or trivial merge).

In Part D, both branches edited the exact same line of results_summary.md after diverging. Git has no way to know which version is “right,” so it pauses the merge and asks you to decide.

Question: While results_summary.md still has conflict markers in it, what does git status say?

TipClick to reveal the answer 
On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")

Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   results_summary.md

Git refuses to let you commit until you resolve and git add the file — this is a safety net so a half-finished conflict can never accidentally become a permanent commit.

Part E — .gitignore for raw data

mkdir raw_data
touch raw_data/run1.fastq raw_data/run2.fastq raw_data/instrument.log
git status     # git sees all these files

Create .gitignore:

nano .gitignore
raw_data/
*.log
git status     # ignored files no longer appear
git add .gitignore
git commit -m "added gitignore for raw instrument data"

What you learned

Command What it does
git branch List branches
git checkout -b <name> Create and switch
git merge <branch> Merge into current branch
resolve + git add + git commit Fix a conflict
.gitignore Exclude files from tracking
Note

Why this matters for real work: raw sequencing files, large datasets, and instrument logs don’t belong in Git history — they’re huge and usually already backed up elsewhere. .gitignore keeps your repository focused on code, scripts, and small text files that benefit from version control.
